메인 커맨드입니다. 목표를 설명하면 relay가 팀을 구성하고, 에이전트들이 병렬로 작업해 완수합니다.

## 사용법

```
/relay:relay "만들거나 수정할 내용 설명"
```

예시:
```
/relay:relay "장바구니 기능을 추가해줘"
/relay:relay "사용자 프로필 페이지를 개선해줘"
/relay:relay "결제 API 보안을 강화해줘"
```

## 실행하면 무슨 일이 일어나나요?

### 1. 팀 구성

에이전트가 실행되기 전, relay가 어떤 종류의 태스크인지 질문하고 풀에서 3~6명의 팀을 제안합니다. 팀을 직접 조정할 수 있습니다 — 전문가를 추가하거나, 불필요한 역할을 빼거나, 같은 역할을 여러 명으로 구성할 수도 있습니다 (예: 서로 리뷰하는 FE 두 명).

확정되면 세션이 시작됩니다.

### 1b. 계획 단계 (선택사항)

태스크가 복잡한 경우, 전체 팀 실행 전에 PM만 먼저 실행하는 경량 사전 실행 단계를 요청할 수 있습니다. PM이 요구사항을 태스크로 분해해두면 모든 에이전트가 빈 태스크 보드 대신 명확한 태스크 보드를 갖고 시작할 수 있습니다. 이 단계는 선택사항 — 전체 팀이 세션 시작 시 함께 계획할 수도 있습니다.

### 2. 모든 에이전트 동시 실행

팀의 모든 에이전트가 동시에 시작합니다. 코디네이터는 원래 요청을 받아 태스크로 분해하고, 나머지 에이전트들은 즉시 메시지와 태스크를 확인합니다 — 순서를 기다리지 않습니다.

각 에이전트는 다음을 가지고 작업합니다:
- `agents.pool.yml`의 역할 페르소나
- 이전 세션의 개인 메모리 (`read_memory`)
- 공유 프로젝트 컨텍스트 (`project.md`)

### 3. 이벤트 드리븐 협업

에이전트들은 MCP 도구를 통해서만 소통합니다 — 직접 에이전트 간 호출은 없습니다:

- `send_message` / `get_messages` — 메시지 전달
- `create_task` / `claim_task` — 원자적 태스크 클레임 (두 에이전트가 같은 태스크를 동시에 가져가지 않음)
- `post_artifact` / `get_artifact` — 산출물 공유
- `request_review` / `submit_review` — 피어 리뷰

리뷰가 필요하면 에이전트가 리뷰어 이름으로 신호를 보냅니다. 리뷰어가 실행되고, 리뷰가 끝나면 원래 에이전트가 재개됩니다. 오케스트레이터 개입 없이 P2P로 진행됩니다.

### 4. 세션 종료

모든 에이전트가 작업을 마치면 세션이 아카이브되고 결과가 리포트됩니다.

## 안정성 기능

### 오케스트레이터 상태 지속

오케스트레이터는 매 반복 후 `save_orchestrator_state`로 이벤트 루프 상태를 저장합니다. 재실행 시(예: 컨텍스트 압축 후) `get_orchestrator_state`를 호출해 전체 루프 상태 — `dormant_agents`, `done_agents`, `last_seen_msg_id` 등 — 를 복원하고, 모든 메시지를 처음부터 재처리하지 않고 정확히 이전 지점에서 재개합니다.

<Aside type="tip">
세션 시작 시 출력되는 대시보드 URL에서 실시간으로 확인할 수 있습니다. 자세한 내용은 [대시보드](/relay/ko-kr/guides/dashboard)를 참고하세요.
</Aside>

## 여러 인스턴스 동시 실행

충돌 없이 여러 relay 인스턴스를 병렬로 실행할 수 있습니다 — 여러 프로젝트를 동시에 작업할 때 유용합니다:

```json
{
  "mcpServers": {
    "relay":   { "env": { "DASHBOARD_PORT": "3456", "RELAY_INSTANCE": "project-a" } },
    "relay-b": { "env": { "DASHBOARD_PORT": "3457", "RELAY_INSTANCE": "project-b" } }
  }
}
```

각 인스턴스는 별도의 포트, 대시보드, 독립된 in-memory 저장소를 갖습니다.
