`/relay:agent` invokes one specific agent in isolation.
Use it when you need a single agent's expertise rather than the full team workflow.

## Usage

```
/relay:agent {agentId} "{instruction}"
```

Examples:
```
/relay:agent fe "Refactor the CartItem component"
/relay:agent be "Security review the user auth API"
/relay:agent qa "Write edge case test scenarios for the login feature"
/relay:agent designer "Write UX improvement proposals for the checkout flow"
```

## Available agent IDs

The available agent list depends on your `agents.yml` configuration.
Check the IDs from the `list_agents` result.

## Execution steps

1. Check available agents via `list_agents`
2. Print dashboard URL: `http://localhost:3456`
3. Load the specified agent's persona and memory
4. Spawn the agent in isolation with its `tools` array restricting MCP tool access
5. On completion, save learnings via `append_memory`

## Memory loading

Single agents work with the full project context:
1. `read_memory(agent_id: "{agentId}")` — the agent's personal memory
2. `read_memory()` — project memory (`project.md` + `lessons.md`)

Both results are merged and prepended to the system prompt.

## Unknown agent handling

If the specified agent ID is not found in `list_agents`:
- Display the list of available agents to the user
- Ask the user to choose a different agent

## When to use

**Single agent invocation is appropriate when:**
- You need a code review of a specific file
- You need architecture analysis or technical advice
- You only need a specific deliverable (tests, docs) without the full workflow
- The task is quick and doesn't justify spinning up the whole team
