# relay — Project Summary

Generated by: init scan (PM cross-validation)

## Product Overview
relay is a multi-agent collaboration framework built as a Claude Code plugin. It simulates a full startup engineering team (PM, Designer, DA, FE, BE, QA, Deployer) where each agent has a distinct role and communicates peer-to-peer via an MCP server. No direct Claude API calls — all AI processing is delegated to Claude Code's Agent tool.

## Tech Stack
- **Runtime**: Bun (NOT Node.js) — bun:sqlite, Bun.serve(), Bun.file()/Bun.write()
- **Language**: TypeScript strict mode
- **Backend**: MCP server (stdio) + Hono HTTP API + Bun WebSocket — all in one process (port 3456)
- **Frontend**: React 19 + Vite 8 + Tailwind CSS v4 (no config file, @tailwindcss/vite plugin)
- **DB**: bun:sqlite (WAL mode, auto-migration, singleton pattern)
- **Memory**: .relay/memory/ Markdown files (cross-session) + SQLite (session-ephemeral)
- **Linter/Formatter**: Biome (replaces ESLint + Prettier)
- **Package manager**: bun workspaces (monorepo)

## Architecture
Three-layer plugin:
1. **MCP Server** — communication infrastructure (message bus, task board, artifact store, memory layer)
2. **Skills** — orchestration strategy (.md files in skills/)
3. **Hooks** — PostToolUse automation → dashboard status push

Packages:
- `packages/server/` (@custardcream/relay) — MCP + Hono + WebSocket
- `packages/shared/` (@custardcream/relay-shared) — AgentId, RelayEvent types
- `packages/dashboard/` (@custardcream/relay-dashboard) — React SPA
- `packages/docs/` (@custardcream/relay-docs) — Astro + Starlight (not yet published)

## Agent Workflow DAG
planning → design → development → review → qa → deploy → _done
(review can loop back to development; qa can loop back to development for bugs)

## MCP Tools (14 total)
- Messaging: send_message, get_messages
- Tasks: create_task, update_task, get_my_tasks
- Artifacts: post_artifact, get_artifact
- Review: request_review, submit_review
- Memory: read_memory, write_memory, append_memory
- Sessions: save_session_summary, list_sessions, get_session_summary
- Orchestrator: list_agents, get_workflow

## DB Schema (5 tables)
messages, tasks, artifacts, reviews, events — all keyed by session_id

## Frontend Architecture
- Global state: useReducer + discriminated union Action (no Redux/Zustand)
- WebSocket: useRelaySocket hook (exponential backoff reconnect, stale-closure-safe via useRef)
- 3-panel layout: TaskBoard (Kanban) | MessageFeed | AgentThoughts
- Dark theme only: gray-950 root, gray-800 cards
- Agent color coding: pm=purple, designer=pink, da=yellow, fe=blue, be=green, qa/deployer=orange

## Test Coverage
- Test runner: bun:test (Jest-compatible)
- 13 test files covering: DB schema, all 5 query modules, all 6 MCP tool handlers, agent loader
- Pattern: in-memory SQLite (":memory:") + runMigrations in beforeEach
- NOT covered: Hono API, WebSocket, React dashboard, E2E scenarios, hooks

## CI/CD
- ci.yml: 4 parallel jobs — Biome check, bun test, tsc --noEmit, dashboard build
- release.yml: Changesets → auto "Version Packages" PR → auto npm publish on merge
- deploy-docs.yml: GitHub Pages on packages/docs/** changes

## Key Conventions
- Comments: Korean (inline), English (variable names, file-path headers)
- Release: always use `bunx changeset` workflow — never run publish directly
- MCP tool naming: snake_case, always include agent_id
- agents.default.yml: never modify — use agents.yml for overrides
- .mcp.json: always specify bin name explicitly with --package flag
- Commit .relay/memory/ to git for shared team memory
