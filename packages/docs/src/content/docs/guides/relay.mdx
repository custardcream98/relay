The main command. Describe a goal ‚Äî relay assembles a team, and they work in parallel until it's done.

## Usage

```
/relay:relay "description of what to build or fix"
```

Examples:
```
/relay:relay "Add a shopping cart feature"
/relay:relay "Improve the user profile page"
/relay:relay "Harden the payment API security"
```

## Zero-config first run

When you run `/relay:relay` on a project without `.relay/agents.pool.yml`, relay automatically:

1. **Analyzes your project** ‚Äî reads README, package.json, directory structure, and config files
2. **Generates a tailored agent pool** ‚Äî selects 4-8 agents across three lanes:
   - **Coordination**: PM (always present)
   - **Implementation**: FE, BE, mobile, devops, etc. based on detected tech stack
   - **Quality**: QA (if test runner detected), security (if auth/payments detected)
3. **Writes `.relay/agents.pool.yml`** ‚Äî customize freely, or delete and re-generate

The generated prompts reference actual file paths, commands, and conventions from your project. After generation, relay proceeds directly to team composition.

<Aside type="tip">
To regenerate the pool: delete `.relay/agents.pool.yml` and run `/relay:relay` again.
</Aside>

## What happens when you run it

### 1. Team composition

Before any agents spawn, relay asks what kind of task this is and suggests a team of 3‚Äì6 agents from your pool matched to the work. You can refine the team ‚Äî add specialists, remove roles that don't apply, or add multiple instances of the same role (e.g. two frontend engineers reviewing each other's work).

Once confirmed, the session starts.

### 1b. Planning Phase (optional)

If the task is complex, you can ask relay to run a lightweight PM-only pre-spawn phase before the full team spawns. The PM breaks down requirements into tasks, so every agent starts with a clear task board instead of an empty one. This is optional ‚Äî the full team can also do planning collaboratively at session start.

### 2. All agents spawn simultaneously

Every agent on the team starts at the same time. The coordinator receives your original request and begins breaking it down. All other agents check for messages and tasks immediately ‚Äî no waiting in line.

Each agent works with:
- Its role persona from `agents.pool.yml`
- Personal memory from previous sessions (`read_memory`)
- Shared project context (`project.md`)

### 3. Event-driven collaboration

Agents communicate exclusively through MCP tools ‚Äî there are no direct agent-to-agent calls:

- `send_message` / `get_messages` ‚Äî messaging
- `create_task` / `claim_task` ‚Äî atomic task claiming (no two agents grab the same task)
- `post_artifact` / `get_artifact` ‚Äî sharing deliverables
- `request_review` / `submit_review` ‚Äî peer review

When an agent needs a review, it signals the reviewer by name. The reviewer spawns, does the review, and the original agent resumes. No orchestrator involvement needed ‚Äî it's peer-to-peer.

### 4. Session end

When all agents finish their work, the session is archived and results are reported to you.

## Reliability features

### Orchestrator state persistence

The orchestrator saves its event loop state after each iteration using `save_orchestrator_state`. On re-spawn (e.g. after context compaction), it calls `get_orchestrator_state` to restore the full loop state ‚Äî `dormant_agents`, `done_agents`, `last_seen_msg_id`, and more ‚Äî and resumes exactly where it left off without replaying all messages from scratch.

<Aside type="tip">
Watch it happen in real time at the dashboard URL printed at session start. See [Dashboard](/relay/guides/dashboard) for details.
</Aside>

## Running multiple instances

You can run several relay instances in parallel without conflicts ‚Äî useful for working on multiple projects at once:

```json
{
  "mcpServers": {
    "relay":   { "env": { "DASHBOARD_PORT": "3456", "RELAY_INSTANCE": "project-a" } },
    "relay-b": { "env": { "DASHBOARD_PORT": "3457", "RELAY_INSTANCE": "project-b" } }
  }
}
```

Each instance gets its own port, dashboard, and isolated in-memory store.
