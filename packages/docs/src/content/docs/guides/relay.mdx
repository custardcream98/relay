The main command. Describe a goal — relay assembles a team, and they work in parallel until it's done.

## Usage

```
/relay:relay "description of what to build or fix"
```

Examples:
```
/relay:relay "Add a shopping cart feature"
/relay:relay "Improve the user profile page"
/relay:relay "Harden the payment API security"
```

## What happens when you run it

### 1. Team composition

Before any agents spawn, relay asks what kind of task this is and suggests a team of 3–6 agents from your pool matched to the work. You can refine the team — add specialists, remove roles that don't apply, or add multiple instances of the same role (e.g. two frontend engineers reviewing each other's work).

Once confirmed, the session starts.

### 2. All agents spawn simultaneously

Every agent on the team starts at the same time. The coordinator receives your original request and begins breaking it down. All other agents check for messages and tasks immediately — no waiting in line.

Each agent works with:
- Its role persona from `agents.pool.yml`
- Personal memory from previous sessions (`read_memory`)
- Shared project context (`project.md`)

### 3. Event-driven collaboration

Agents communicate exclusively through MCP tools — there are no direct agent-to-agent calls:

- `send_message` / `get_messages` — messaging
- `create_task` / `claim_task` — atomic task claiming (no two agents grab the same task)
- `post_artifact` / `get_artifact` — sharing deliverables
- `request_review` / `submit_review` — peer review

When an agent needs a review, it signals the reviewer by name. The reviewer spawns, does the review, and the original agent resumes. No orchestrator involvement needed — it's peer-to-peer.

### 4. Session end

When all agents finish their work, the session is archived and results are reported to you.

<Aside type="tip">
Watch it happen in real time at the dashboard URL printed at session start. See [Dashboard](/relay/guides/dashboard) for details.
</Aside>

## Running multiple instances

You can run several relay instances in parallel without conflicts — useful for working on multiple projects at once:

```json
{
  "mcpServers": {
    "relay":   { "env": { "DASHBOARD_PORT": "3456", "RELAY_INSTANCE": "project-a" } },
    "relay-b": { "env": { "DASHBOARD_PORT": "3457", "RELAY_INSTANCE": "project-b" } }
  }
}
```

Each instance gets its own port, dashboard, and isolated in-memory store.
