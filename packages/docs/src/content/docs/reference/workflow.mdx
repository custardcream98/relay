## Schema

```typescript
interface WorkflowConfig {
  jobs: Record<string, JobConfig>;
}

interface JobConfig {
  agents: string[];                      // agent IDs to run in parallel
  description: string;                   // job description (injected into agents)
  reviewers?: Record<string, string[]>;  // reviewer mapping
  end: Record<string, string>;           // nextJobId → condition description
}
```

## Execution mechanism

### Start job detection

```
Build the set of all destination job IDs from every end map
→ The job not in this set = the start job
```

### Job execution flow

```
1. Spawn job.agents in parallel
2. Inject into each agent:
   - Agent persona (list_agents)
   - Personal memory (read_memory(agent_id))
   - Project memory (read_memory())
   - Current job info (description + end conditions)
3. Agents work, then declare end:
   send_message(to: null, content: "end:{nextJobId} | {reason}")
4. Collect all agents' end declarations
5. Determine next job → repeat
```

### Disagreement handling

If agents declare different `nextJobId` values:
- The orchestrator (Claude) reads the `end` condition descriptions and the collected reasons, then makes the final decision

### On reaching `_done`

1. Each agent saves session learnings via `append_memory`
2. Update team `lessons.md`
3. Call `save_session_summary`
4. Report results summary to the user

## End declaration format

```
"end:{nextJobId} | {reason}"
```

Examples:
```
"end:qa | All PRs have received LGTM."
"end:development | Auth header handling needs a fix."
"end:_done | Deployment complete and team notified."
```

## reviewers mechanism

```yaml
review:
  agents: [fe2, be2]
  reviewers:
    fe: [fe2]
    be: [be2]
```

When spawning the `fe2` agent:
  1. Load the `fe` agent's persona (fe2 is another instance of fe)
  2. Set `agent_id` to `fe2`
  3. Inject additional context: "Retrieve and review the fe-pr artifact created by fe via get_artifact"

## Constraints

- Job IDs must be unique
- `_done` is a reserved keyword; use it only as a final destination
- Circular references are not supported (however, non-circular back-edges such as `review → development → review` are allowed)
- The `agents` array in a job must have at least one entry

## Default values (agents.default.yml)

```yaml
workflow:
  jobs:
    planning:
      agents: [pm]
      description: "Analyse user requirements and break down into tasks."
      end:
        design: "When all tasks have been created and assigned"

    design:
      agents: [designer, da]
      description: "Designer writes the UX spec; DA writes the analytics plan. Both must complete."
      end:
        development: "When both deliverables have been shared via post_artifact"

    development:
      agents: [fe, be]
      description: "FE implements the UI; BE implements the API. Both create PR artifacts."
      end:
        review: "When implementation is complete and PRs are ready"

    review:
      agents: [fe2, be2]
      reviewers:
        fe: [fe2]
        be: [be2]
      description: "Review the PRs from quality, security, and performance perspectives."
      end:
        qa: "When all PRs are LGTM"
        development: "When changes are requested"

    qa:
      agents: [qa]
      description: "Test the feature and create tickets for any bugs found."
      end:
        deploy: "When all tests pass and there are no open bug tickets"
        development: "When bugs are found and fixes are needed"

    deploy:
      agents: [deployer]
      description: "Deploy the QA-approved feature to production."
      end:
        _done: "When deployment is complete and team is notified"
```
